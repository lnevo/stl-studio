STL Studio – Help & Reference

A visual editor for Tower LCC+Q logic programming

🚀 Quick Start ⚡ How RLO Works 🔵 Logic blocks 🟤 Nesting blocks 🟢 Output blocks 🟠 Timer blocks 🟡 Jump blocks 📖 Tutorials

What is STL Studio?

STL Studio is a visual, block-based editor for writing STL (Statement List) logic programs for the Tower LCC+Q — a 16-channel I/O board that adds PLC-style logic to any NMRA LCC layout.

Instead of typing STL by hand (which is error-prone), you drag and connect blocks. The editor generates the exact STL text you paste into your Tower LCC+Q node via JMRI's Configure Nodes → Conditionals → Logic.

The Tower LCC+Q implements a subset of the Siemens S7 STL language. It supports only boolean (true/false) operations — no math, no counters, no analog values. But that's all you need for signals, crossings, interlockings, and most layout automation.

Quick Start

  1. Open the editor at lnevo.github.io/stl-studio
  2. Load your node's variable names (optional but recommended): click Import Config… and paste your node's config backup. The editor will populate friendly names for I/Q/M/Y/Z variables.
  3. Drag blocks from the left panel into the workspace. Blocks snap together vertically — the order they appear top-to-bottom is the order they execute.
  4. Read the STL output in the right panel — it updates live as you edit. Turn on Split view for a side-by-side raw + plain-English explanation.
  5. Copy to clipboard and paste into JMRI: LCC → Configure Nodes → [your node] → Conditionals → Logic (group N) → MultiLine. Click Write then reboot the node.
Tip: The node stores up to 16 logic groups, each with 256 characters. The byte count bar in the editor shows you how full each group is.

Right panel — STL output

  • Readable format adds spaces between operands for readability.
  • Compact format omits all spaces — required if you're close to the 256-byte group limit.
  • Split view: top pane shows raw STL; bottom pane shows each instruction with a plain-English explanation as a comment.
  • Show line numbers: adds a line-number gutter. Numbers skip comment/blank lines.
  • Byte count bar: tracks how many of the 256 characters are used. Goes amber near the limit, red if over.

Importing existing STL

Click Import STL…, paste your existing STL text (one statement per line, or compact inline), then click Import into workspace. The importer supports all standard instructions including compact, multi-statement lines.

Variables & Addresses

All variables are boolean (true/false only). There are no integers, no analog values, no strings.

PrefixNameRangeDirectionNotes

QLogic OutputQ0.0 – Q15.7 (128)Write-only from logicChanges produce LCC events. Cannot be changed by the network (only by logic).

ZTrack TransmitterZ0.0 – Z15.7WriteSends track-circuit speed codes to adjacent masts.

Caution: You cannot = (assign), S (set), or R (reset) an I or Y variable — they are inputs only. You cannot send LCC events directly to a Q variable — it is output-only.
Tip: Group addresses as x.y where x is 0–15 (group) and y is 0–7 (bit within group). So I3.5 = input group 3, bit 5.

How STL Works — RLO & Logic Strings

Understanding the RLO (Result of Logic Operation) is the key to writing correct STL. Think of the RLO as a single true/false value that flows through your instructions like water through a pipe.

The logic string

A logic string is a sequence of bit-logic instructions (A, O, AN, …) that builds up an RLO step by step, followed by a terminating instruction that uses or writes the result.

  • The first bit instruction in a string sets the RLO from its variable — it is not combined with any previous RLO.
  • Each subsequent bit instruction combines its variable with the current RLO using the operator (AND, OR, etc.).
  • A terminating instruction (=, S, R, JC, CLR, SET, or a timer start) consumes the RLO and resets it so the next bit instruction starts a fresh string.
A   I1.0    // start new string: RLO = I1.0
AN  I1.1    // AND with NOT I1.1
O   M0.3    // OR with M0.3
=   Q2.0    // write RLO to Q2.0; starts new string
=   M0.0    // can assign same RLO to multiple outputs

NOT does not terminate

NOT simply inverts the RLO in place — it does not end the logic string. The next bit instruction will still combine with (the now-inverted) RLO.

The /FC bit

Behind the scenes the node tracks a /FC (Not First Check) bit. When /FC = 0 the next bit instruction starts a fresh logic string. When /FC = 1 it combines with the running RLO. Terminating instructions reset /FC to 0. You never set this directly — it's automatic.

SAVE and the BR register

SAVE copies the current RLO into the BR (Binary Result) register without terminating the logic string. This lets you carry a result forward across terminating instructions. The JCB and JNB jump instructions also set BR.

Program Cycle

The Tower LCC+Q re-evaluates all logic every 20 ms in a continuous loop. This means:

  • Logic outputs (Q) fire LCC events only when their value changes — not on every scan.
  • An input event that appears and disappears in under 20 ms may be missed entirely.
  • Jumps are forward-only. The logic never loops back (no infinite loops possible).
  • The logic executes up to 16 groups × 256 characters = 4,096 total characters per cycle.
Note: Timer instructions run off a separate hardware clock, not the 20 ms logic scan. A timer set for 500 ms will fire after 500 ms regardless of how many scan cycles pass.

Logic (RLO) — Boolean Conditions

These blocks check the state of a variable and combine the result with the running RLO.

BlockSTLWhat it does
ANDA <var> If var is ON (1), AND it with the current RLO. Both must be true for the result to be true.
AND NOTAN <var> Check that var is OFF (0), then AND with RLO. True when the variable is not active.
ORO <var> If var is ON, OR it with the current RLO. Either being true is enough.
OR NOTON <var> Check that var is OFF, then OR with RLO.
Exclusive ORX <var> XOR: true when exactly one of the two (RLO or var) is true. Chaining multiple X instructions: true if an odd number are true.
Exclusive OR NOTXN <var> XOR NOT: like X but tests whether var is OFF.
NOTNOT Inverts (flips) the RLO. Does not terminate the logic string.
CLRCLR Forces the RLO to 0 (false). Starts a new logic string.
SETSET Forces the RLO to 1 (true). Starts a new logic string.
SAVESAVE Saves the current RLO into the BR register without ending the string. Used to carry a result past a terminating instruction.

Example — simple AND condition

A   I0.0    // Block occupancy sensor is active
AN  I0.1    // AND facing-point lever is NOT in position
=   Q1.0    // Set signal to Stop

Nesting — Grouped Sub-expressions

Nesting works like parentheses in algebra — it lets you group a set of conditions and combine the group's result with the outer logic. Up to 7 levels of nesting are allowed.

BlockSTLWhat it does
[A] AND ( … )A( … )Evaluate the group; AND its result with the outer RLO.
[AN] AND NOT ( … )AN( … )Evaluate the group; AND NOT its result with the outer RLO.
[O] OR ( … )O( … )Evaluate the group; OR its result with the outer RLO.
[ON] OR NOT ( … )ON( … )Evaluate the group; OR NOT its result with the outer RLO.
[X] XOR ( … )X( … )Evaluate the group; XOR its result with the outer RLO.
[XN] XOR NOT ( … )XN( … )Evaluate the group; XOR NOT its result with the outer RLO.

In the editor, each nesting block is a C-shaped container. Drop logic blocks inside the container; they are evaluated as a unit.

Example — route cleared when one of two routes is set AND no opposing train

O(
  A  I1.0    // Route A lever
  A  I1.1    // Route A clear
O(
  A  I1.2    // Route B lever
  A  I1.3    // Route B clear
)
)
AN I2.0     // AND no opposing train
=  Q3.0     // Route cleared

Outputs — Statement Termination

These instructions consume the current RLO, write its value to a variable, and reset the RLO so the next instruction starts a fresh logic string.

BlockSTLWhat it does
Assign == <var> Copies the RLO directly to the variable every scan. If RLO is true, var is true; if RLO goes false, var follows immediately. Multiple = blocks in a row all receive the same RLO.
Latch ON SS <var> Sets (latches) the variable to 1 when RLO = 1. The variable stays 1 even after the condition becomes false — until explicitly reset with R.
Latch OFF RR <var> Resets the variable to 0 when RLO = 1. Works on Q, M variables and also on timers (R Tn stops and clears a timer).
Key difference: = tracks the condition continuously. S/R only change the output when the RLO first becomes 1 — they create a memory effect. Use S/R pairs when you need something to stay on even after the activating condition clears (e.g. an alarm latch).

Example — assign vs latch

// Assign: signal follows sensor directly
A   I0.0
=   Q0.0

// Latch: set alarm when sensor fires; reset by button
A   I0.1    // sensor
S   M1.0    // latch alarm flag ON
A   I0.2    // reset button
R   M1.0    // clear alarm flag

Transition — Edge Detection

Edge instructions detect the moment the RLO changes state and produce a one-scan-cycle pulse (20 ms). They require a memory variable to store the previous state for comparison.

BlockSTLWhat it does
FP (rising edge)FP <Mx.x> Sets RLO = 1 for exactly one scan cycle when the RLO transitions from 0 → 1. Stores the previous state in Mx.x.
FN (falling edge)FN <Mx.x> Sets RLO = 1 for exactly one scan cycle when the RLO transitions from 1 → 0. Stores the previous state in Mx.x.
Important: The memory variable used by FP/FN is dedicated to that edge detector. Do not use the same M address for two different FP/FN instructions, and do not use it for anything else — it stores internal state, not a real logic value.

Example — pulse output for one cycle when a sensor first activates

A   I2.0    // occupancy sensor
FP  M2.0    // detect rising edge; store flag in M2.0
=   Q2.0    // Q2.0 true for exactly 20ms on first activation

Timers

The Tower LCC+Q has 64 hardware timers, T0–T63. Timers must always be preceded by a Load (L) instruction that sets the time value.

Loading a timer value

The format is L W#t#xyz where:

  • t = time base: 0 = 10 ms, 1 = 100 ms, 2 = 1 s, 3 = 10 s
  • xyz = time value (decimal, up to 999)

Examples: L W#0#10 = 100 ms  |  L W#1#5 = 500 ms  |  L W#2#30 = 30 s

Timer types

BlockSTLBehaviour
SD On-delayL … / SD Tn Timer starts when RLO = 1. Output (T bit) goes true after the delay, stays true as long as RLO stays 1. Resets immediately when RLO goes 0.
SE Extended pulseL … / SE Tn Timer starts on rising edge of RLO. Output stays true for the full preset time, regardless of whether the input (RLO) goes false. Retriggers if RLO goes true again during the pulse.
SP PulseL … / SP Tn Like SE but output goes false if RLO goes false before the time expires.
SS Retentive on-delayL … / SS Tn Like SD but the elapsed time is retained if RLO goes false. Timing resumes from where it stopped when RLO goes true again. Must be explicitly reset with R Tn.
SF Off-delayL … / SF Tn Output goes true immediately when RLO goes 1. When RLO goes 0, output stays true for the preset time, then goes false. Useful for a "hold-off" after a condition clears.
FR Enable/FreeFR Tn Re-enables a running timer so it can be restarted. Not needed for initial start — only use FR when you need to re-trigger an already-running timer.
R Reset timerR Tn Clears the timer and resets its output. Required to clear an SS timer. Also works on Q and M.
Tip: After a timer start instruction (SD, SE, etc.), the RLO is reset — the next instruction starts a new logic string. To check if a timer has finished, use A Tn (the T bit goes true when the timer has timed out).
Caution: Never use timer names like T1 or T23 in comments. The node's compiler scans comments and may misinterpret them as timer references.

Example — SD on-delay (signal holds green for 5 seconds after train clears)

A   I1.0         // block occupancy
L   W#2#5         // load 5 seconds
SD  T1            // start on-delay timer T1
A   T1            // check if timer has expired (new logic string)
=   Q1.0          // output true after 5s of continuous occupancy

Jumps & Labels

Jump instructions let you skip over sections of logic based on the current RLO. Jumps are forward-only — you cannot jump backwards (no loops possible by design).

A jump target is defined by a Label block (e.g. E1:). Labels must be 1–4 alphanumeric characters followed by a colon, and must appear at the start of a line. Labels cannot be in the middle of a logic string.

BlockSTLWhen does it jump?
JU Jump alwaysJU <label>Always — regardless of RLO.
JC Jump if trueJC <label>When RLO = 1 (condition is true). RLO is set to 1 after the jump (whether taken or not).
JCN Jump if falseJCN <label>When RLO = 0 (condition is false).
JCB Jump if true + BRJCB <label>When RLO = 1. Also copies RLO into the BR register.
JNB Jump if false + BRJNB <label>When RLO = 0. Also copies RLO into the BR register.
JBI Jump if BR=1JBI <label>When the BR register = 1 (set by a previous SAVE/JCB/JNB).
JNBI Jump if BR=0JNBI <label>When the BR register = 0.

In the editor, the C-shaped jump block automatically encloses everything between the jump and its matching label. The label appears at the bottom of the C-shape.

Example — IF / THEN / ELSE pattern

// IF block occupied THEN set Stop, ELSE set Clear
A   I0.0
JCN APPR        // skip to APPR if NOT occupied
SET              // occupied: force RLO true
=   Q1.0         // Stop aspect
JU  DONE
APPR:
CLR
=   Q1.0         // Clear aspect
DONE:

Groups & Comments

Logic group header

The [//Group N:] block marks the beginning of a logic group. Place one at the very top of each group's stack. The number is calculated automatically from its position in the workspace. You can give the group a descriptive name (e.g. "East Signal", "Crossing Gate").

Each group corresponds to one of the 16 MultiLines in the Tower LCC+Q node. Each MultiLine can hold up to 256 characters.

Comment

The [//] block inserts a // comment line. Comments are stripped by the node's compiler and do not count toward byte usage. Use them freely to explain your logic.

Caution: Do not put timer names (T0–T63) inside comments. The compiler scans comment text and may confuse timer names in comments with actual timer instructions.

Tutorial 1 — Simple Signal Logic

Goal: Set a signal to Stop (Q1.0 = 1) when a block is occupied (I0.0 = 1) or the facing-point lever is wrong (I0.1 = 0). Otherwise set Clear (Q1.0 = 0).

Step 1 — Think through the logic

Stop if: block occupied OR lever not set. Clear otherwise. Since the output follows the condition directly, use =.

Step 2 — Build the blocks

  1. Drag a Logic → OR block, set variable to I0.0.
  2. Drag an OR NOT block below it, set variable to I0.1.
  3. Drag an Outputs → Assign (=) block, set variable to Q1.0.

Resulting STL

O   I0.0    // block occupied
ON  I0.1    // OR lever NOT in position
=   Q1.0    // Stop signal
Tip: The first instruction is O (OR), not A. Since it's the first instruction in the string, it just sets RLO = I0.0. The second instruction ON then ORs with that value.

Tutorial 2 — Delayed Output (On-Delay Timer)

Goal: Activate a relay output (Q2.0) only after a block has been occupied for at least 3 seconds — to debounce brief detections.

Step 1 — Plan

Use an SD (on-delay) timer. The timer output (T-bit) goes true after 3 s of continuous input.

Step 2 — Build the blocks

  1. Drag a Logic → AND block, variable = I0.0.
  2. Drag a Timers → Load (L) block, set value = W#2#3 (3 seconds).
  3. Drag a Timers → On-delay (SD) block, timer = T0.
  4. Drag a Logic → AND block, variable = T0.
  5. Drag an Outputs → Assign (=) block, variable = Q2.0.

Resulting STL

A   I0.0       // occupancy sensor
L   W#2#3       // load 3 seconds
SD  T0          // start on-delay timer (terminates string)
A   T0          // new string: check if timer has finished
=   Q2.0        // relay output
Note: The SD instruction terminates the first logic string. A T0 starts a brand new string that checks whether the timer has timed out.

Tutorial 3 — Latch / Memory (S and R)

Goal: Latch an alarm flag (M0.0) when any of three sensors fires. Clear the flag when a reset button (I3.0) is pressed.

Resulting STL

// Set alarm flag if any sensor fires
O   I0.0
O   I0.1
O   I0.2
S   M0.0    // latch ON — stays true even when sensors clear

// Clear alarm flag on button press
A   I3.0
R   M0.0    // latch OFF

// Drive output from flag
A   M0.0
=   Q3.0    // alarm indicator

Tutorial 4 — Grouped Conditions (Nesting)

Goal: Protect a junction — allow route only if (Route A set AND A-path clear) OR (Route B set AND B-path clear), AND no opposing train detected.

Resulting STL

O(
  A  I1.0   // Route A lever
  A  I1.1   // Route A path clear
)
O(
  A  I1.2   // Route B lever
  A  I1.3   // Route B path clear
)
AN I2.0     // AND no opposing train
=  Q3.0     // Route permitted

The two O( … ) groups each evaluate independently, then their results are OR'd together, then AND NOT'd with the opposing train sensor.

Byte Limits

  • Each logic group (MultiLine) holds 256 characters max.
  • There are 16 logic groups, for a total of 4,096 characters.
  • Comments are stripped at compile time and do not count toward the 256-character limit.
  • Use Compact output format to save bytes (removes all spaces).
  • The byte count bar turns amber at 90% capacity and red if you exceed 256 characters.
  • Spaces between the operator and variable count. A I0.0 = 6 bytes; AI0.0 = 5 bytes.

Tips & Gotchas

  • First instruction in a string is special. The very first bit instruction after a terminating operation starts a new logic string — it sets the RLO from scratch, not combines. This means O I0.0 as a first instruction acts exactly like A I0.0.
  • Multiple = assignments. You can chain several = Q… blocks one after another — they all receive the same RLO value. Useful for driving multiple outputs from one condition.
  • Label length. Jump labels must be 1–4 alphanumeric characters. E1, STOP, AB1 are valid. STOP1 (5 chars) is not.
  • No label in the middle of a string. Jump targets must appear at the beginning of a statement, not in the middle of a logic chain.
  • Timer names in comments. Avoid writing T0, T1, …, T63 anywhere in a comment. The compiler's pre-processor scans comment text and may accidentally interpret them as timer references.
  • FP/FN memory variable. Each FP or FN instruction needs its own dedicated M variable. Using the same M for two edge detectors or for other logic will cause incorrect results.
  • Jumps are forward-only. You cannot create loops. This is a deliberate safety feature of the Tower LCC+Q implementation.
  • 20 ms update cycle. Very brief LCC events (shorter than 20 ms) may be missed by the logic. Space your sensors accordingly at high train speeds.
  • Q can be read back. A Q variable you assign in one group can be read in another group on the same node using A Q…, etc. — treating it as an internal flag.
  • Reboot to compile. After writing new logic via JMRI, you must reboot the node (Configure Nodes → More… → Reboot) to trigger the compile cycle. Check Syntax Messages 1–3 after the reboot for errors.
